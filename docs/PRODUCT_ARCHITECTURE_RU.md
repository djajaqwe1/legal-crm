# Архитектура продукта: две стороны, ЛК клиента, мультитенант для юристов

Документ фиксирует **поправку к ТЗ**: продукт — не «одна админка», а связка **медиа-лицо → лидогенерация → личный кабинет клиента → рабочее место юриста**, с возможностью **тиражирования** (каждый покупатель — свой тенант со своим лендингом и настройками).

---

## 1. Три слоя ценности

| Слой | Для кого | Задача |
|------|-----------|--------|
| **Публичная часть** | Посетитель из рекламы / поиска | Доверие, бренд, запись на консультацию, сбор лида |
| **Личный кабинет клиента** | Уже заинтересованный человек | Ограниченный диалог с AI-«консультантом», материалы, **воронка**: «продолжим у юриста (Стёма)» — запись/оплата слота |
| **Кабинет юриста (CRM)** | Юрист и команда | Дела, документы, задачи, клиенты из воронки, полноценная работа |

Опционально четвёртый слой: **оператор платформы** (вы как вендор) — биллинг, лимиты, поддержка, модерация.

---

## 2. Две разные сессии входа

- **Клиент портала** — отдельная учётная запись (e-mail + magic link или пароль), cookie/сессия **не смешивается** с CRM.
- **Сотрудник (юрист)** — вход в workspace своей фирмы; внутри — роли `owner` / `lawyer` / `assistant` (см. `lib/product-zones.ts`).
- **Текущий парольный вход в `/admin`** — временный «монолит»; целевое состояние: **поддомен или путь тенанта** + нормальная аутентификация (лучше OAuth2/OIDC позже).

ИИ-консультант в ЛК клиента: **жёсткие лимиты** (сообщений в день, токены, только FAQ/общие вопросы), явный дисклеймер «не юридическая консультация», CTA «записаться к юристу».

---

## 3. Мультитенантность «для тиража»

**Рекомендуемая модель для SaaS:** один деплой приложения, в БД у каждой сущности **`workspaceId`** (тенант = купивший юрист/контора).

- **Изоляция данных:** все запросы Prisma фильтруются по `workspaceId` из сессии; супер-админ платформы — отдельная роль без фильтра только для саппорта.
- **Публичный сайт:** `https://{slug}.yourdomain.com` или `https://yourdomain.com/u/{slug}` — контент и тема из `WorkspaceSettings` (JSON: цвета, логотип, тексты блоков, ссылка на запись).
- **Лендинг под себя:** не «переписать код», а **настройки + блоки** (hero, услуги, отзывы, CTA, контакты); при необходимости — кастомный CSS в рамках гайдлайна бренда платформы.

Альтернатива «отдельный VPS на юриста» — проще по изоляции, но **дороже** в поддержке и обновлениях; для масштаба лучше **один код + tenant в БД**.

---

## 4. Воронка «AI → человек»

1. Лендинг → форма / виджет.
2. ЛК или гостевой чат с лимитом → квалификация (тип вопроса, регион).
3. Кнопка **«Записаться на консультацию к Стёму»** → календарь / заявка → в CRM создаётся **лид + задача юристу**.
4. В CRM юрист видит источник (канал, UTM), историю AI-чата (read-only), следующий шаг.

---

## 5. Этапы реализации (порядок работ)

1. **Модель Workspace + User** в Prisma, привязка существующих `Client` / `LegalCase` к `workspaceId` (миграция + сид для Conglomerate Altai).
2. **Разделение маршрутов:** `/` маркетинг, `/portal/*` ЛК клиента, `/app/*` или `/w/[slug]/admin` CRM; `proxy` по разным cookie/ролям.
3. **ЛК клиента:** регистрация/вход, чат с квотой, хранение сообщений по `portalUserId`, экран «запись на консультацию».
4. **Настройки лендинга** в UI для owner тенанта (без деплоя кода).
5. **Биллинг и лимиты** (подписка, количество юристов, AI-кредиты).

Текущий код остаётся основой **слоя 3**; слои 1–2 наращиваются поверх и через общую модель лидов.
